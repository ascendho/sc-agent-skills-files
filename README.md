## 一、Agent Skills 是什么

### 1. 核心定义 (核心观点)

- **Agent Skills 是什么：** 它们是一种**轻量级、开放格式**的标准，用于扩展 AI 智能体的功能。
- **物理形态：** 一个“技能”本质上是一个**包含组织化文件的文件夹**。
- **功能：** 它包含了指令、脚本、资产（如图片、图标）和资源，智能体可以通过“发现”这些文件夹，从而更准确地执行特定任务。

### 2. 架构流程 (右侧图示)

课件通过右侧的流程图解释了智能体如何获取能力：

1. **Agent（智能体）：** 主体程序。
2. **Filesystem（文件系统）：** 存储技能的地方。
3. **Skill（技能包）：** 文件系统中存储了多个不同的技能文件夹。
4. **交互：** 智能体指向文件系统，读取其中的“技能”文件夹，从而获得执行该特定任务所需的全部素材和逻辑。

### 3. 三个具体的技能示例 (左侧文件夹结构)

这三张图展示了不同类型的技能是如何组织的，体现了“技能”的多样性：

#### **示例 1：营销活动分析 (Analyzing Marketing Campaign)**

- **重点：** 规则与参考。
- **结构：** 包含 SKILL.md（核心指令）和一个 references/ 文件夹，里面有“预算重新分配规则 (budget_reallocation_rules.md)”。
- **用途：** 侧重于**决策依据和业务逻辑**。

#### **示例 2：PDF 处理 (PDF)**

- **重点：** 自动化脚本与工具。
- **结构：** 除了说明文件，还包含一个 scripts/ 文件夹，里面有多个 Python 脚本（如：检查可填写表单、将 PDF 转为图像、提取表单信息等）。
- **用途：** 侧重于**技术执行和程序化处理**。

#### **示例 3：时事通讯设计 (Designing Newsletters)**

- **重点：** 设计素材与模板。
- **结构：** 包含 references/（风格指南）、assets/（页眉图片、图标）以及 templates/（HTML 模板、Word 文档布局）。
- **用途：** 侧重于**内容创作、视觉表现和标准化输出**。

### 总结

这页课件传达的核心信息是：**Agent 的能力不应该只是写在一段 Prompt（提示词）里，而应该像软件插件一样，封装在一个结构化的文件夹中。**

这种“技能包”模式的优点在于：

- **模块化：** 方便开发者复用和分享技能。
- **易于维护：** 脚本、资产和说明文档分开存放，清晰明了。
- **可扩展性强：** 只要往文件系统里丢一个新的文件夹，Agent 就能学会一项新本事。

## 二、为什么需要 Agent Skills

### 1. 过去的设计思路：垂直领域的专用智能体 (左侧)

- **模式：** 以前我们倾向于为每个特定领域开发一个独立的智能体。**Coding Agent**（编程智能体）：专门写代码。**Research Agent**（研究智能体）：专门检索信息。**Marketing Agent**（营销智能体）：专门做策划。**Finance Agent**（财务智能体）：专门分析报表。
- **弊端：** 这种方式导致了大量的重复建设。每个智能体都是一个独立的“孤岛”，如果你想让一个营销智能体写一点代码，它可能就做不到了。

### 2. 现代的设计思路：通用智能体 + 代码接口 (右侧)

- **核心观点：代码是万能接口（Code is the universal interface）。**
- **模式：** 我们不再构建无数个专用智能体，而是构建一个**通用的智能体（General-Purpose Agent）**。这个智能体最核心的能力是**调用代码（LLM + Code Interpreter）**。
- **逻辑：** 只要这个智能体能运行 Bash 脚本、Python 代码，并能操作文件系统（Filesystem），它理论上就能完成各种任务（如访问网页、处理文档、生成演示文稿、分析财务、管理日程等）。

### 3. 为什么还需要“Skills”？ (底部的文字)

虽然通用智能体很强大，但它面临一个现实问题：

- **简单脚手架（Simple Scaffolding）：** 仅仅给智能体一个 Bash 环境和文件系统是不够的。这就好比给了一个聪明的工人一套完整的工具箱，但他不知道你公司的具体业务流程。
- **核心矛盾：** 智能体需要 **“上下文（Context）”** 和 **“领域专业知识（Domain Expertise）”** 才能**可靠地（Reliably）**完成工作。

### 总结：Agent Skills 的角色

这页课件实际上给出了 **Agent Skills** 的存在理由：

1. **从“专用”转向“通用”：** 我们不需要为每个任务写一个复杂的智能体程序。
2. **技能包作为“插件”：** 既然代码是通用接口，那么我们将某个领域的“专业知识、参考资料、标准作业程序（SOP）和专用脚本”打包成一个 **Skill（即前几页课件提到的文件夹）**。
3. **赋予灵魂：** 当通用智能体“加载”了某个 Skill 文件夹时，它就瞬间获得了该领域的专业背景，从而能更专业、更可靠地执行任务。

## 三、为什么需要 Agent Skills II

### 1. 核心价值：按需加载的知识与上下文

第一页课件指出，**Skills（技能）** 为智能体提供了：

- **程序化知识（Procedural knowledge）：** 告诉智能体“如何做”的具体步骤。
- **特定上下文（Specific context）：** 包含公司、团队或用户特有的背景信息。
- **按需加载（Load on demand）：** 智能体不需要预先训练所有知识，而是像查阅手册一样，在需要时从文件系统中读取对应的技能包。

### 2. 三大核心维度与应用场景

课件将技能的价值分为三个维度，并给出了具体例子：

#### **A. 领域专家知识 (Domain Expertise)**

- **场景：** 品牌指南与模板、法律审查流程、数据分析方法论。
- **作用：** 让通用大模型瞬间拥有像“资深设计师”或“法务专家”一样的专业水准，遵循特定的业务标准。

#### **B. 可重复的工作流 (Repeatable Workflow)**

- **场景：** 每周营销活动回顾、客户电话准备流程、季度业务回顾（QBR）。
- **作用：** 针对那些周期性、有固定套路的工作。通过技能包，智能体可以每次都以同样的高标准完成任务，确保流程的一致性。

#### **C. 扩展新能力 (New Capabilities)**

- **场景：** 制作幻灯片（PPT）、生成 Excel 表格或 PDF 报告、构建 MCP（Model Context Protocol）服务器。
- **作用：** 通过技能包里的脚本（如 Python 脚本），智能体可以完成它原生 LLM 无法直接做到的技术性输出。

### 3. 反面对比：如果没有 Skills 会怎样？

第二页右侧的方框总结了“传统模式”或“仅靠提示词（Prompting）”的缺陷：

- **重复性工作：** 你不得不每次都重新向智能体描述你的指令和详细要求。
- **手动准备文件：** 每次都要手动打包、上传所有的参考资料和辅助文件，非常低效。
- **一致性难以保证：** 很难确保智能体每次输出的结果和遵循的流程都是完全一致的，容易产生偏差。

### 总结

这两页课件的中心思想是：**Agent Skills 是将“通用智能”转化为“商业生产力”的关键桥梁。**

- **智能体（Agent）** 是大脑和手。
- **技能（Skill）** 是专业手册、工具箱和标准操作程序（SOP）。
- **结果：** 这种结合让智能体不再是一个只会“聊天”的 AI，而是一个能听懂公司规矩、会用公司工具、能持续稳定产出专业结果的“数字员工”。

## 四、Skills 是可移植的

### 1. 核心观点：一次编写，到处运行

- **跨平台复用：** 开发者在一个地方创建的技能包（Skill），可以在任何兼容该标准的智能体（Agent）中直接使用。
- **无需重复开发：** 你不需要为不同的产品或环境重新编写逻辑。同一个技能文件夹既可以给开发者工具用，也可以给网页端应用用。

### 2. 具体的应用场景示例

课件列举了 Anthropic 生态中的几个典型例子，说明同一个“技能”可以被以下产品共享：

- **Claude Code：** 终端命令行工具（面向开发者）。
- **Claude.ai：** 网页端交互界面（面向普通用户）。
- **Claude Agent SDK：** 开发者用来构建自定义智能体的软件开发工具包。
- **Claude API：** 直接通过代码接口调用的底层能力。

**这意味着：** 如果你开发了一个“自动化生成周报”的技能，你可以在终端里运行它，也可以在网页聊天框里调用它，甚至可以把它集成到你自有的 App 中。

### 3. 行业标准与开放性

- **开放标准（Open Standard）：** Agent Skills 不仅仅是某个产品的私有功能，而是一个正在被推广的开放标准。
- **生态兼容性：** 课件提到已有越来越多的智能体产品开始采用这一标准。这暗示了未来不同厂商、不同架构的 AI 智能体之间，能力是可以互通和交换的。

### 4. 图示解读

- **中间的 "Skill" 方框：** 代表了前几页提到的结构化技能文件夹。
- **两侧的 Agent：** 左边是 **Claude Code**，右边是基于 **Claude Agent SDK** 构建的自定义智能体。
- **箭头指向：** 两个完全不同的智能体环境都在“读取”同一个技能包。这形象地展示了技能作为“独立插件”存在，而不与特定的智能体程序耦合。

### 总结

这页课件传达的最重要信息是 **“解耦”** 和 **“标准化”**：
**Agent Skills 让 AI 的能力变成了像“手机 App”或“U 盘”一样的东西——只要接口对得上，插在哪儿都能用。** 这极大地降低了 AI 应用的开发成本，并促进了智能体能力的快速流通。

## 五、Skills 是可组合的

### 1. 核心观点：乐高式的能力组装

- **可组合性：** 技能包不是孤立存在的，它们可以像积木一样拼在一起。
- **复杂工作流：** 智能体可以同时加载多个技能，并按照逻辑顺序调用它们，从而完成一个靠单一提示词（Prompt）很难做到的复杂任务。

### 2. 工作流示例：从数据到汇报的自动化

课件通过四个具体的技能展示了一个完整的业务链路：

1. **BigQuery Skill (数据查询技能)：**作用： 提供与营销相关的数据库表结构（Schema）。*智能体第一步：知道去哪里取数，以及数据长什么样。*
2. **Marketing campaign analysis skill (营销活动分析技能)：**作用： 对取回的营销数据进行深入分析。*智能体第二步：发挥“大脑”作用，挖掘数据背后的意义和趋势。*
3. **Company brand skill (公司品牌技能)：**作用： 提供品牌的视觉指南（如：字体、颜色、Logo 等）。*智能体第三步：获取公司规范，确保输出的内容符合品牌形象。*
4. **PowerPoint skill (PPT 技能)：**作用： 将分析结果和品牌元素结合，自动生成演示文稿（Slide Deck）。*智能体第四步：执行最终的交付物产出。*

### 3. 这种组合模式的优势

- **模块化：** 每个技能都是独立的。如果你想把 BigQuery 换成 Snowflake 数据库，只需要替换第一个“积木”即可。
- **关注点分离：**数据工程师维护数据库技能；分析师维护分析逻辑技能；设计师维护品牌指南技能。**智能体则负责把这些人的专业知识整合在一起。**
- **高度定制：** 不同的部门可以根据自己的需求，从“技能库”里挑选不同的技能进行组合，形成自己专属的工作流。

### 总结

这页课件告诉我们：**Agent Skills 的强大之处不在于单个技能有多全能，而在于它们可以被灵活地编排。**

通过这种方式，智能体可以处理像“分析上周广告表现并生成一份符合品牌视觉要求的汇报 PPT”这样跨领域、跨工具的复杂商业任务。

## 六、Skills 是如何工作的

### 1. 三个关键特性

- **信息容量大 (Skills can contain a lot of information)：**
  由于技能是以文件夹形式存储在文件系统中的，它可以包含大量的文档、长篇指南、复杂的脚本和资源。这些内容如果全部塞进提示词（Prompt），会远远超出 LLM 的处理能力。
- **海量扩展性 (You can have hundreds of skills)：**
  系统设计支持同时存放成百上千个技能。这意味着智能体可以拥有一个庞大的“技能库”，涵盖从财务分析到代码调试等各行各业的能力。
- **渐进式披露 (Progressive Disclosure) —— 核心技术手段：**
  这是本页最重要的技术点。课件指出：**为了保护上下文窗口（Context Window），技能是“渐进式披露”的。**

### 2. 什么是“渐进式披露 (Progressive Disclosure)”？

在 AI 领域，这是一个非常聪明的策略。它的工作原理类似于：

1. **索引阶段：** 智能体最初并不会“阅读”所有技能文件夹里的所有内容。它只知道有哪些技能可用（类似于看书的目录）。
2. **按需调取：** 只有当任务真正需要某个特定技能时，智能体才会去文件系统中打开对应的文件夹。
3. **局部读取：** 即使打开了文件夹，它也可能只读取其中某个特定的脚本或一段规则，而不是把整个文件夹里的东西一次性吞掉。

**为什么要这么做？**

- **节省空间：** LLM 的上下文窗口（即它能同时记住的信息量）是有限且昂贵的。如果一次性加载几百个技能，窗口会瞬间爆满，导致智能体“失忆”或报错。
- **提高效率：** 只关注当前任务相关的知识，能让智能体运行得更快、更精准。

### 3. 图示解读

- **Agent（智能体）：** 右侧的方框代表智能体，它由 LLM（大脑）和执行环境（代码运行环境）组成。
- **Filesystem（文件系统）：** 中间的方框是存储技能的地方。
- **交互箭头：** 智能体指向文件系统。这代表智能体拥有**操作文件系统的权限**。它不是“死记硬背”所有知识，而是在需要时，通过代码接口去文件系统里“翻书”。

### 总结

这页课件揭示了 Agent Skills 的底层逻辑：**它将“长期记忆/专业知识（存储在文件系统中的技能）”与“短期思考空间（LLM 的上下文窗口）”进行了解耦。**通过**渐进式披露**，智能体可以变成一个“博学多才”的专家，同时依然保持思考的敏捷和准确，因为它只在需要时才去调用那部分海量的专业技能。

## 七、渐进式披露

### 1. 第一层：元数据 (Metadata) —— “始终加载”

- **内容：** 顶部蓝色阴影区域（YAML Frontmatter 格式）。
- **关键信息：** 技能名称（name）和功能描述（description）。
- **逻辑：** 智能体**始终**会将这部分信息保存在记忆中。
- **作用：** 这相当于技能的“名片”或“摘要”。当用户提出问题时，智能体通过扫描所有技能的 description，来判断当前是否需要调用这个“营销活动分析”技能。

### 2. 第二层：核心指令 (Instructions) —— “触发时加载”

- **内容：** 下方深蓝色区域（Markdown 格式）。
- **关键信息：**输入（Input）： 期待的数据列（Date, Spend, Revenue 等）。**指标计算（Metrics）：** CTR、CVR、ROAS、CPA、净利润的计算公式。**输出格式（Output Tables）：** 规定了结果表格的样式。
- **逻辑：** 只有当智能体确定要使用这个技能时，才会把这部分详细的“操作手册”加载到当前的对话上下文中。
- **作用：** 告诉智能体执行任务的具体步骤和计算逻辑。

### 3. 第三层：资源 (Resources) —— “按需加载”

- **内容：** 右侧橙色方框代表的外部文件（如 budget_reallocation_rules.md）。
- **关键信息：** 每周优化框架、具体的预算调整规则（例如：当 ROAS 超过目标 15% 时，增加 10-15% 的预算）。
- **逻辑：** 即使技能被触发了，智能体也不会立刻读取这些细碎的规则。只有当任务进行到“需要根据规则做预算决策”这一步时，智能体才会去文件系统里“翻阅”这个特定的资源文件。
- **作用：** 存放深度专业知识或庞大的规则库，避免干扰智能体的核心逻辑思考。

### 总结：为什么要设计这三层加载机制？

通过这种**金字塔式**的信息披露结构，Agent Skills 解决了大模型的一个核心矛盾：

1. **省钱省空间：** 不把成千上万行的规则一次性塞给 AI。
2. **防止干扰：** 如果一次性给太多信息，AI 容易在复杂的计算中“迷路”（注意力分散）。
3. **极度专业：** 这种结构允许一个技能包里包含极度详尽的业务规则（如：复杂的法律条款、繁杂的财务公式），只要在最终决策阶段按需调用即可，保证了输出的高度专业性和一致性。

**一句话总结：** 这种机制让智能体表现得像个职场高手——**平时只记住每个工具箱是干嘛的，用的时候才打开说明书，遇到疑难杂症才去翻阅背后的专业参考手册。**

## 八、通用智能体

### 1. 核心层：智能体（The Agent）

位于中心的大方框是主体，它包含三个核心要素：

- **LLM（大语言模型）：** 智能体的“大脑”，负责理解指令、推理和决策。
- **代码执行能力 (**</></>**)：** 智能体不仅仅是聊天，它还能编写并运行代码（如 Python、Bash），这是它与物理世界交互、处理数据的“手”。
- **Tools（工具）：** 智能体内置或可以直接调用的特定功能函数。

### 2. 知识与逻辑层：文件系统与技能（Filesystem & Skills）

- **位置：** 左侧。
- **作用：** 这就是前面课件重点介绍的内容。
- **逻辑：** 智能体通过访问文件系统，按需加载不同的 **Skill（技能包）**。这为智能体提供了特定领域的专业知识、标准作业程序（SOP）和参考资料。

### 3. 连接层：MCP 服务器（MCP Servers）

- **位置：** 右侧（MCP 代表 Model Context Protocol，模型上下文协议）。
- **作用：** 智能体通过 MCP 协议连接到外部服务。
- **示例：** 这里的服务器可能连接着 Google Drive、GitHub、Slack 或公司内部数据库。
- **逻辑：** 如果说 Skill 是智能体学到的“知识”，那么 MCP Server 就是智能体使用的“外部设备”和“实时数据源”。

### 4. 协作层：子智能体（Subagents）

- **位置：** 底部。
- **作用：** 分身与分工。
- **逻辑：** 对于极其复杂的任务，主智能体不需要亲力亲为。它可以“雇佣”或创建多个**子智能体（Subagents）**，将任务拆解并分发下去。例如，一个子智能体负责找资料，另一个负责写代码，第三个负责审查结果。

### 总结：通用智能体的工作模式

这页课件传达了一个核心愿景：**未来的 AI 不再是一个简单的对话框，而是一个“数字指挥官”。**

1. **理解：** 通过 LLM 理解用户的模糊需求。
2. **装备：** 从 Filesystem 加载相关的 **Skill**（获得专业知识）。
3. **联网：** 通过 **MCP Server** 调用外部工具和实时数据。
4. **执行：** 运行代码处理数据，或者指挥 **Subagents** 协同作战。
5. **交付：** 完成复杂的端到端任务。

**一句话总结：**
这套架构让智能体实现了**“大脑（LLM）+ 专业手册（Skills）+ 工具连接（MCP）+ 团队协作（Subagents）”**的完美结合，从而能够处理真正复杂的现实业务。

## 九、 Skills 对比 MCP

### 1. Skills (左侧)：逻辑与知识

- **核心定义：** Skills 告诉智能体**如何处理数据**（Skills teach your agent what to do with that data）。
- **示例：** 图中左侧方框写着“使用这张表的 A 列和 B 列计算指标 X”。
- **特点：**它是一套**指令和逻辑**。它像是一本**“操作手册”**或**“ SOP（标准作业程序）”**。它存在于智能体可以访问的**文件系统**中。
- **本质：** 赋予智能体“软实力”，即专业领域的方法论。

### 2. MCP (右侧)：连接与数据源

- **核心定义：** MCP 将智能体**连接到外部系统和数据**（MCP connects your agent to external systems and data）。
- **结构：** 智能体通过不同的 MCP 服务器（MCP server 1, 2, 3）连接到外部的 **Database（数据库）**、API 或其他生产力工具。
- **特点：**它是一套**基础设施和连接协议**。它像是智能体的**“管道”**、**“眼睛”**或**“手”**，让它能触达公司内部私有数据。它解决了“数据在哪里”以及“如何安全取回数据”的问题。
- **本质：** 赋予智能体“硬连接”，即获取实时信息和操作外部工具的能力。

### 3. 两者如何协同工作？（核心逻辑）

这页课件通过架构图展示了一个完整的闭环：

1. **取数 (via MCP)：** 智能体通过 **MCP** 从数据库中提取出原始的营销数据表。
2. **加工 (via Skill)：** 智能体从 **Filesystem** 中读取“营销分析 **Skill**”，学习到“要把 A 列和 B 列相除来计算转化率”这样的业务逻辑。
3. **产出：** 智能体结合“拿到的数据”和“学到的逻辑”，最终输出一份准确的分析报告。

### 4. 形象的类比

为了方便理解，你可以这样记忆：

- **MCP 是“自来水管”：** 它把水（数据）从远处的各种水库（数据库/系统）引到你家里。
- **Skills 是“咖啡机”：** 它是一套机器和程序，告诉智能体如何利用这些水（数据）来煮出一杯好喝的咖啡（业务结果）。

### 总结

这页课件的目的是消除开发者的困惑：有了 MCP 这种强大的连接协议，为什么还需要 Agent Skills？
**结论是：** MCP 解决了“连接”问题，但无法解决“业务逻辑标准化”问题。只有将两者结合，才能构建出一个既能读懂私有数据，又能像资深员工一样按规矩办事的专业智能体。

## 十、Skills 对比 Tools

### 1. 工具 (Tools)：基础的原子能力

- **定义：** 工具为智能体提供完成任务所需的**基本、必需的能力**。
- **存储位置：** 工具的定义（名称、描述、参数）**始终存在于上下文窗口（Context Window）中**。这意味着智能体随时都知道这些工具怎么用。
- **示例 (以 Claude Code 为例)：**文件操作：Read (读), Write (写), Edit (编辑)。系统交互：Bash (运行命令行), Glob/Grep (搜索文件)。外部获取：WebFetch (抓取网页), WebSearch (搜索网页)。
- **本质：** 它们是智能体与物理世界交互的“原语”或“基础动词”。

### 2. 技能 (Skills)：专业的知识扩展

- **定义：** 技能通过**专业化的知识**来扩展智能体的能力边界。
- **存储位置：** 存储在外部**文件系统（Filesystem）**中。
- **组成：** 就像左图所示，一个技能（如 pdf/）是一个文件夹，里面包含说明文档 (SKILL.md)、参考资料 (reference.md) 和具体的脚本 (scripts/)。
- **特点：**按需工具 (Tools on demand)： 技能可以包含脚本，当智能体加载该技能时，这些脚本就变成了临时可用的“工具”。**不占用常驻内存：** 只有在需要处理特定任务（如“填充 PDF 表单”）时，技能才会被读取。
- **本质：** 它是针对特定领域（如 PDF 处理、营销分析）封装的一整套“专业手册 + 专用工具”。

### 3. 核心区别总结

| 维度         | 工具 (Tools)                   | 技能 (Skills)                          |
| ------------ | ------------------------------ | -------------------------------------- |
| **定位**     | 基础、通用的底层能力           | 专业、特定的领域知识                   |
| **资源消耗** | **始终占用**上下文窗口         | **按需加载**，保护上下文窗口           |
| **形态**     | 预定义的函数接口               | 包含指令和脚本的文件夹                 |
| **类比**     | 像是一个人的“手、眼、基本常识” | 像是一个人书架上的“专业教材和专用软件” |

### 4. 两者如何协同工作？

课件通过中间的架构图展示了它们的协作：

1. **Agent（智能体）** 内部预装了一套 **Tools**（如运行 Python 的能力）。
2. 当遇到复杂任务时，智能体从 **Filesystem** 中读取 **Skill**。
3. 智能体使用它内置的 **Tools**（比如 Bash 或 Python 解释器）去运行技能文件夹里的 **Scripts**（专业脚本）。

**一句话总结：**
**工具（Tools）** 赋予了智能体“能干活”的基础，而 **技能（Skills）** 赋予了智能体“干好特定专业活”的智慧和资源。

## 十一、Skills 对比 Subagents

### 1. 技能 (Skills)：静态的专业支撑

- **定义：** 技能为智能体提供专业知识、特定语言或框架的最佳实践。
- **存储：** 存放在文件系统（Filesystem）中，作为一种静态资源存在。
- **通用性：** 技能可以提供给**主智能体（Main Agent）**，也可以提供给其下属的**任何一个子智能体（Subagent）**。
- **类比：** 技能就像是一个书架上的“专业操作指南”或“专用插件”。

### 2. 子智能体 (Subagents)：动态的独立执行者

- **定义：** 主智能体可以创建专门的子智能体来处理特定的任务。
- **核心特性：**独立性： 子智能体拥有自己**独立的上下文（Context）和工具权限**。这意味着它们在处理任务时不会干扰主智能体的思考空间。**委派机制：** 主智能体将任务“委派”给子智能体，子智能体独立完成工作后，仅将**最终结果**返回给主智能体。
- **示例：** 如图中所示的 **Code Reviewer（代码审查员）**。主智能体可以分出一个专门负责审代码的子智能体。
- **类比：** 子智能体就像是一个公司里的“部门主管”或“外包专家”。

### 3. 两者的协作关系 (图中的关键连线)

课件中有一个非常重要的细节：**紫色箭头从 Filesystem 里的某个 Skill 指向了 Subagent。**

- **逻辑：** 当主智能体创建一个“代码审查子智能体”时，它可以给这个子智能体配备一个特定的技能包（例如：包含特定编程语言最佳实践的技能）。
- **意义：** 这样，子智能体就变成了一个**“带薪自带工具和知识库的专家”**。它在独立的环境中，利用专业的技能包完成任务，既高效又安全。

### 总结

这页课件展示了复杂 AI 系统的分工哲学：

- **Skills（技能）解决“怎么做才专业”的问题。**
- **Subagents（子智能体）解决“任务如何拆解和隔离执行”的问题。**

**最终效果：** 主智能体像是一个指挥官，它不需要自己精通所有细节，而是通过给不同的“专业人员（Subagents）”发放对应的“操作手册（Skills）”，来驱动整个复杂工作流的运转。

## 十二、整合一切 (Putting Everything Together)

### 1. 中心枢纽：智能体 (The Agent)

这是整个系统的“指挥部”，它由三个核心部分驱动：

- **LLM (大语言模型)：** 提供理解、推理和决策能力（大脑）。
- **代码执行环境 (**</></>**)：** 让智能体能够运行 Python 或 Bash 脚本，将想法转化为实际操作（手）。
- **Tools (基础工具)：** 智能体常驻内存的基础原子能力（如读写文件、搜索等）。

### 2. 知识扩展层：技能 (Skills via Filesystem) —— 位于左侧

- **角色：** 它是智能体的**“专业参考资料库”**。
- **协同逻辑：** 当智能体面临特定领域的复杂任务（如法律审查或复杂的 PDF 处理）时，它会从文件系统中查找并读取对应的 **Skill**。
- **价值：** 通过“渐进式披露”机制，它让智能体在不撑爆记忆（上下文窗口）的前提下，拥有无限扩展的专业知识。

### 3. 数据连接层：MCP 服务器 (MCP Servers) —— 位于右侧

- **角色：** 它是智能体连接外部世界的**“管道”**。
- **协同逻辑：** 智能体通过 **MCP 协议** 安全地触达公司数据库、GitHub、Slack 或云端服务。
- **价值：** 解决了“数据在哪里”的问题，让智能体能够实时获取并操作外部系统中的数据。

### 4. 任务执行层：子智能体 (Subagents) —— 位于底部

- **角色：** 它是智能体的**“分身或下属”**。
- **协同逻辑：** 主智能体将复杂的总任务拆解，指派给拥有独立上下文和特定权限的**子智能体**去处理。
- **价值：** 实现了任务的并行处理和上下文隔离，防止主智能体在处理海量细节时产生混乱。

### 总结：一个完整的业务链路示例

为了理解这张图的强大之处，我们可以想象智能体处理一个任务的过程：

1. **取数：** 智能体通过 **MCP 服务器** 从数据库中调取上周的销售数据。
2. **学逻辑：** 智能体从 **Filesystem** 加载“财务分析 **Skill**”，学习公司特有的利润计算公式。
3. **拆任务：** 主智能体派出一个 **Subagent**，专门负责使用代码工具（**Tools**）将这些数据生成图表。
4. **汇总：** 主智能体将结果汇整，通过代码工具生成一份最终的 PDF 报告。

**最终结论：**
这套架构通过 **Skills（逻辑）**、**MCP（数据）**、**Tools（能力）** 和 **Subagents（协作）** 的完美配合，将 AI 从一个简单的“聊天机器人”变成了一个真正能够胜任复杂、专业、企业级任务的**数字员工**。

## 十三、客户洞察分析器

### 1. 数据接入：通过 MCP 获取原始资料

- **操作：** 智能体利用右侧的 **Google Drive MCP 服务器**。
- **目标：** 连接到公司云端的文件夹，获取两类核心数据：**客户访谈笔记 (Customer interview notes)**：通常是非结构化的文本。**调查问卷回复 (Survey responses)**：通常是结构化或半结构化的数据。

### 2. 逻辑加载：通过 Skills 获得专业方法论

- **操作：** 智能体从左侧的 **Filesystem** 中加载特定的 **Skill（技能包）**。
- **内容：** 这个技能包里包含了一份**“如何对反馈进行分类以及如何总结调研发现的指南”**。
- **作用：** 确保智能体不仅是在“读”数据，而且是按照公司认可的、专业的分析标准来进行“审读”和“分类”。

### 3. 任务委派：通过 Subagents 进行分工协作

- **操作：** 主智能体（Agent）根据数据类型，将任务拆分给两个专门的子智能体：**访谈分析员 (Interview Analyzer)：** 专门负责利用加载的“技能”去深入挖掘访谈记录中的客户痛点。**调查分析员 (Survey Analyzer)：** 专门负责统计和分析问卷结果。
- **优势：** 每个子智能体都有独立的上下文空间，处理各自擅长的领域，最后将结果汇总给主智能体。

### 4. 核心大脑：智能体 (Agent) 的指挥作用

- **角色：** 位于中心的 Agent 负责调度一切。它通过 **LLM** 进行逻辑推理，决定什么时候去取数（MCP），什么时候查手册（Skill），以及如何分配任务（Subagents）。
- **工具辅助：** 它还会利用内置的 **Tools** 和代码执行能力（`</></>`）来进行数据计算或格式化输出。

### 总结：这个案例证明了什么？

这个“客户洞察分析器”不再只是一个简单的提示词工程，而是一个**企业级的智能系统**：

1. **标准化：** 所有的分析逻辑都封装在 **Skill** 里，换个 Agent 运行效果也一样。
2. **实时性：** 通过 **MCP** 直接读取最新数据，不需要手动复制粘贴。
3. **可扩展：** 如果以后增加了“社交媒体评论分析”，只需增加一个新的 Subagent 和对应的 Skill 即可。

**一句话总结：**
这页课件展示了如何利用这套架构，将繁杂的原始客户数据，通过标准化的专家逻辑（Skill）和高效的分工（Subagents），自动转化为高质量的商业洞察。

## 十四、总结对比表

### 1. Skills (技能) —— 专业的“操作手册”

- **提供什么：** 程序化知识（如何一步步完成专业任务）。
- **持久性：** **跨对话持久**。它以文件形式存在，不随对话结束而消失。
- **包含内容：** 指令 + 代码脚本 + 静态资产（如模板、图片）。
- **加载时机：** **动态加载，按需调用**（不占用不必要的内存）。
- **最适用于：** **特定领域的专业知识**（如财务分析、PDF处理）。

### 2. Prompts (提示词) —— 临时的“口头指令”

- **提供什么：** 即时的、当下时刻的指令。
- **持久性：** **仅限单次对话**。对话结束，记忆清除。
- **包含内容：** 纯自然语言描述。
- **加载时机：** **每一轮对话**都会加载。
- **最适用于：** **快速、简单的请求**（如“帮我总结这段话”）。

### 3. Subagents (子智能体) —— 委派的“专职人员”

- **提供什么：** 任务委派（将大任务拆交给小助手）。
- **持久性：** **跨会话持久**。子智能体被设计为可以独立运行并返回结果。
- **包含内容：** 完整的智能体逻辑（大脑+工具+权限）。
- **加载时机：** **被主智能体召唤时**加载。
- **最适用于：** **高度专业且需要隔离的任务**（如代码审查、独立的调研任务）。

### 4. MCP (模型上下文协议) —— 连接的“管道”

- **提供什么：** 工具和系统的连接性。
- **持久性：** **持续连接**。它是基础设施层面的打通。
- **包含内容：** 工具定义（即 API 接口描述，告诉 AI 有哪些外部功能可用）。
- **加载时机：** **始终可用**。一旦配置好，智能体随时可以调用。
- **最适用于：** **访问外部数据源**（如数据库、Google Drive、GitHub）。

### 核心差异总结：

| 组件          | 核心解决的问题             | 关键词                 |
| ------------- | -------------------------- | ---------------------- |
| **Skills**    | “如何专业地做这件事？”     | **专业方法论、脚本包** |
| **Prompts**   | “你现在要帮我做什么？”     | **即时指令、自然语言** |
| **Subagents** | “这个复杂子任务谁来负责？” | **任务隔离、分工协作** |
| **MCP**       | “数据在哪，我怎么拿到它？” | **外部连接、数据获取** |

**最终结论：**
一个强大的 AI 智能体通常是这四者的结合：通过 **Prompts** 接收用户指令，通过 **MCP** 获取数据，通过 **Skills** 学习处理逻辑，必要时召唤 **Subagents** 来协作完成。课件底部还附带了一个链接（[claude.com/blog/skills-explained](https://www.google.com/url?sa=E&q=https%3A%2F%2Fwww.claude.com%2Fblog%2Fskills-explained)），供想要深入研究的开发者查阅官方文档。

1. **工具化开发：** 提到 skill-creator，说明创建 Skill 是有工具支撑的，不是纯手写。
2. **从手动到自动化：** 通过 MCP 取代 CSV 上传，体现了企业级应用的进化。
3. **模块化力量：** 开发者不需要写一个巨大的“全能代码”，而是分别维护“分析逻辑”、“品牌资源”和“绘图工具”三个小模块。
4. **业务落地：** 最终交付物（PPT 汇报）是一个真实的商业需求，证明了这套架构在实际办公场景中的巨大潜力。

**简而言之，这两页课件是告诉学员：我们将亲手做一个能自动查数据库、看品牌手册并最后帮你写好 PPT 的“数字营销助手”。**

## 十六、SKILL.md 的内部结构

它是智能体（Agent）理解并执行特定任务的“说明书”。整个文件采用了典型的“文档即代码”的设计思想，分为两个主要部分：

### 1. Frontmatter

- **格式：** 位于文件顶部，由 --- 包裹的 **YAML** 格式数据。
- **核心字段：**name: 技能的唯一标识符（如 analyzing-marketing-campaign）。**description:** 极其重要的部分。它描述了技能的用途、适用场景以及能解决什么问题。
- **作用：** 这是技能的“索引”。智能体在扫描技能库时，主要通过阅读这里的描述来判断：**“当前这个任务，我是不是该用这个技能？”**。

### 2. Body Content（主体内容区）

- **格式：** 采用标准的 **Markdown** 格式。
- **核心内容：** 它是具体的“操作指南”，在图中展示了以下几个维度：**Input（输入）：** 定义了执行该任务所需的数据结构。例如，它要求 Excel/CSV 必须包含日期、渠道、花费、收入等特定列。**Metrics（计算逻辑）：** 明确了业务指标的计算公式。*漏斗指标：* CTR、CVR 的算法。*效率指标：* ROAS（目标≥4.0x）、CPA（最高$50）、净利润计算公式。**Output Tables（输出规范）：** 规定了智能体分析完成后，应该以什么样的表格格式呈现结果。**Budget Reallocation（外部引用）：** 这是一个**“渐进式披露”**的典型例子。它告诉智能体：如果你需要做预算调整决策，请去翻阅 references/ 文件夹下的具体规则文件。

### 总结：这种结构的巧妙之处

1. **易于理解：** 对于人类开发者来说，这就是一份清晰的文档；对于 AI 来说，它是结构化的指令。
2. **解耦决策与执行：**Frontmatter 让智能体决定“要不要做”。**Body Content** 告诉智能体“怎么去做”。
3. **标准化：** 所有的技能都遵循这种结构，使得不同的开发者可以方便地共享和复用技能包。

**一句话总结：**
SKILL.md 是技能包的“灵魂”，它通过 **YAML 元数据** 实现自动发现，通过 **Markdown 指令** 实现逻辑规范，从而让通用智能体能够像专业人士一样处理特定业务。

## 十七、Frontmatter 的具体字段规范

Frontmatter 是智能体识别、发现并决定是否调用该技能的关键依据。它分为**必填字段**和**选填字段**。

### 第一页：必填字段 (Required Fields)

这部分规定了技能最基础、最核心的信息及其严格的约束条件：

#### **1. name (技能名称)**

- **长度限制：** 最多 64 个字符。
- **格式要求：** 只能包含**小写字母、数字和连字符 (-)**。不能以连字符开头或结尾。
- **目录一致性：** **必须与该技能所在的父文件夹名称完全一致**。
- **命名建议：** 推荐使用**动名词（verb + -ing）形式**，例如示例中的 analyzing-marketing-campaign。这有助于智能体将其理解为一个“动作”或“能力”。

#### **2. description (功能描述)**

- **长度限制：** 最多 1024 个字符。
- **基本要求：** 不能为空。
- **内容指南：**应同时描述“技能做什么”以及“什么时候应该使用它”。应包含特定的关键词，帮助智能体精准识别相关的任务需求。这是实现“自动发现技能”的核心。

### 第二页：选填字段 (Optional Fields)

这部分字段用于提供额外的管理信息或运行限制，增加技能的复用性和安全性：

#### **1. license (许可协议)**

- 指明该技能包的授权协议名称或指向许可文件的引用，便于合规性管理。

#### **2. compatibility (兼容性)**

- 最多 500 个字符，用于说明该技能运行所需的环境要求（例如特定的 Python 版本或硬件要求）。

#### **3. metadata (自定义元数据)**

- 允许开发者添加任意的“键-值对”（Key-Value pairs）。
- **示例：** 课件蓝框中展示了 author: example-org 和 version: "1.0"。这对于技能的版本控制和来源追踪非常有用。

#### **4. allowed-tools (允许的工具 - 实验性功能)**

- 这是一个空格分隔的列表，列出该技能预先批准使用的工具。这是一种安全机制，用于限制智能体在执行该技能时能动用的权限。

### 总结：为什么要制定这些规范？

1. **机器可读性：** 严格的 name 命名规范确保了文件路径和逻辑引用的一致性。
2. **智能路由：** 高质量的 description 让大模型（LLM）能像查阅索引一样，在几百个技能中迅速找到最匹配的那一个。
3. **标准化：** 这种 YAML 格式的元数据让 Agent Skills 成为一种可以跨平台、跨项目流通的资产。

**一句话总结：**
Frontmatter 是给 AI 看的“名片”和“使用说明摘要”，必填项确保了**可发现性**，选填项则增强了**可维护性**和**安全性**。

## 十八、Body Content 的编写指南

### 第一页：实践指南 (Practical Guidance)

这部分侧重于如何编写高质量、高性能的指令文档：

- **格式建议：** 虽然没有强制格式限制，但强烈建议包含**分步骤指令**、**输入/输出格式示例**以及**常见边缘情况（异常处理）**。
- **核心准则：**保持精简（Under 500 lines）： 建议长度控制在 500 行以内。这是为了节省智能体的“上下文窗口”，防止由于指令太长导致智能体遗忘或反应迟钝。**模块化引用：** 将详细的参考资料移动到独立文件中。在 SKILL.md 中只显示基础内容，通过链接指向高级或特定领域的详细内容。**浅层结构：** 外部引用建议只保持**一层深度**（即：references/file.md），避免深层嵌套的目录结构。**路径规范：** 统一使用**正斜杠 (/)** 作为文件路径分隔符，确保在不同操作系统（包括 Windows）上的兼容性。

### 第二页：自由度与复杂流 (Degrees of Freedom)

这部分探讨了如何根据任务性质，通过指令控制智能体的发挥空间：

#### **1. 三种自由度 (Degrees of Freedom)**

- **高自由度 (High freedom)：**仅提供通用的文本方向。智能体可以自主选择多种有效方法。*适用于：创意写作、发散性思考任务。*
- **中等自由度 (Medium freedom)：**提供伪代码、示例或模式（Pattern）。有推荐的路径，但也允许智能体根据实际情况进行调整。*适用于：通用的数据处理或分析任务。*
- **低自由度 (Low freedom)：**明确引用特定的脚本，必须遵循特定的执行序列。智能体几乎没有自由裁量权，必须“照方抓药”。*适用于：财务合规计算、法律审查等不容出错的严谨任务。*

#### **2. 复杂工作流的处理 (Complex Workflows)**

- **顺序化：** 将复杂操作拆解为清晰、顺序化的步骤。
- **解耦：** 如果工作流变得非常庞大且步骤繁多，应将其推送到独立的文件中，保持主技能文件的清爽。

### 总结：Body Content 的核心哲学

这两页课件传达了一个非常重要的设计思想：**SKILL.md 应该是一个“精干的调度员”，而不是“臃肿的百科全书”。**

1. **保护上下文窗口：** 通过限流（500行）和模块化，把宝贵的“大脑内存”留给真正的思考。
2. **按需控制：** 开发者可以根据任务的严肃程度，灵活调节给 AI 的“自由度”。
3. **标准化协作：** 结构化的 Body Content 确保了即使是复杂的业务逻辑，也能被通用智能体稳定、可靠地执行。

## 十九、可选子目录

### 1. /scripts（脚本目录）：行动能力

这里存放的是智能体可以运行的代码（如 Python、Bash 脚本）。

- **核心要求：**记录依赖： 明确说明运行该脚本需要安装哪些库或环境。**详尽文档：** 脚本内部或旁边应有清晰的注释，解释其功能。**显式错误处理：** 脚本应该有完善的报错机制（Error Handling），并提供有用的错误信息，这样当脚本运行失败时，智能体能明白发生了什么并尝试修复，而不是直接卡死。
- **关键提示：** 在 SKILL.md 的指令中，必须明确告诉智能体：**这个脚本是用来“执行（Execute）”以获取结果的，还是仅仅作为“参考（Reference）”来阅读其逻辑的。**

### 2. /references（参考目录）：知识储备

这里存放的是供智能体在需要时查阅的补充文档。

- **核心要求：**按需阅读： 这些文件不是一开始就加载的，只有当任务涉及相关细节时，智能体才会去读。**内容聚焦：** 每个参考文件应只关注一个特定话题，保持短小精悍，便于智能体快速消化。
- **关键提示：** 如果一个参考文件超过 **100 行**，强烈建议在文件顶部包含一个**目录（Table of Contents）**。这能让智能体在不读完整个文件的情况下，先看清文件的“全貌”和结构，从而精准定位所需信息。

### 3. /assets（资产目录）：辅助资源

这里存放的是任务中需要用到的各种静态模板、多媒体素材或静态数据。

- **包含内容：**模板（Templates）： 文档模板（如报表格式）、配置文件模板。**图像（Images）：** 流程图、公司 Logo、图标等（用于生成 PPT 或网页）。**数据文件（Data files）：** 查找表（Lookup tables）、数据库表结构定义（Schemas）。

### 总结：为什么要这样分类存放？

这种目录结构体现了 **“高内聚、低耦合”** 的设计思想：

1. **保护上下文窗口：** 复杂的逻辑（脚本）和庞大的知识（参考）被移出了主指令文件。
2. **模块化：** 开发者可以独立更新脚本或参考资料，而不需要频繁修改核心的 SKILL.md。
3. **赋予 Agent 专家身份：**通过 /scripts 赋予 Agent **“手”**（执行力）；通过 /references 赋予 Agent **“大脑”**（专业知识）；通过 /assets 赋予 Agent **“素材库”**（交付能力）。

**一句话总结：**
这三个目录让 SKILL.md 能够保持“精干”，同时让智能体在面对复杂任务时拥有完善的工具箱和图书馆。

## 二十、评估 Skills

### 1. 结构化的单元测试框架

课件通过两个 JSON 格式的示例，展示了评估一个技能时需要定义的四个核心要素：

- **Skills (待测技能)：** 明确指定要测试哪一个具体的技能文件夹。
- **Queries (测试查询/指令)：** 模拟用户可能提出的各种具体请求。
- **Files (测试文件)：** 提供不同格式、不同内容的输入文件（如 .pdf, .tex, .csv），测试技能的鲁棒性。
- **Expected Behavior (预期行为/成功标准)：** 这是评估的“标准答案”，列出了智能体必须满足的一系列具体要求。

### 2. 两个具体的测试案例对比

#### **案例一：生成练习题技能 (Slide 1)**

- **测试重点：** 格式兼容性与内容提取。
- **预期行为标准：**技术准确性： 处理 PDF 时是否正确使用了 pdfplumber 库。**内容完整性：** 是否提取了所有的学习目标？是否生成了 4 种类型的题目？**格式合规性：** LaTeX 输出是否能成功编译？文件名是否正确（名为 output）？

#### **案例二：时间序列分析技能 (Slide 2)**

- **测试重点：** 逻辑流与执行可靠性。
- **预期行为标准：**脚本执行： 内部的 Python 脚本是否运行成功？**工作流顺序：** 步骤顺序是否正确（先分析再画图）？**智能解读：** 对发现的分析结果解读是否正确？**产出管理：** 是否创建了结果文件夹并存入所有必要文件？

### 3. 评估的核心原则 (底部的 "Also" 部分)

除了自动化的单元测试，课件还强调了两个至关重要的补充手段：

1. **获取人工反馈 (Get Human Feedback)：**
   AI 的输出质量（如解读是否深刻、题目是否合理）往往具有主观性，需要人类专家进行最终把关，通过“人在回路”不断优化 SKILL.md 中的指令。
2. **全模型测试 (Test with all the models you're planning to use)：**
   不同的底层模型（如 Claude 3.5 Sonnet vs. GPT-4o）对同一份技能指令的理解能力不同。如果你计划在多个模型上部署，必须在每个模型上都跑一遍测试，确保技能具有跨模型的兼容性。

### 总结：为什么要这么麻烦做评估？

这两页课件传达的信息是：**Agent Skills 的开发已经从“调教提示词”转向了“工程化开发”。**

- **可预测性：** 通过明确的“预期行为”，把模糊的 AI 表现量化为可考核的指标。
- **回归测试：** 当你修改了技能里的某个脚本或规则后，重新跑一遍这些测试，可以确保“没改坏”原有的功能。
- **生产环境保障：** 只有通过了这一系列严苛测试的技能，才能放心地部署到真实业务场景中。
